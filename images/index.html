Red-Black Tree Invariants

What is a Red-Black Tree?

A self-balancing BST that guarantees logarithmic height O(log n).

Balance is maintained by assigning a Color (Red or Black) to every node and enforcing strict structural rules.

Why it matters?

  • Guarantees O(log n) for Search, Insert, Delete.
  • Prevents worst-case skewing.

The 5 Invariants

  • 1.
    Node Color: Every node is either Red or Black.
  • 2.
    Root Property: The root is always Black.
  • 3.
    Null Black: All null leaves are Black.
  • 4.
    Red Constraint: No two Red nodes can be adjacent (Parent & Child cannot both be Red).
  • 5.
    Black Height: Every path from a node to null descendants has the same number of Black nodes.

Operations & Rebalancing

When to fix the tree?

Insertion

New nodes are always RED.

If Parent is Red → Violation! (Double Red)

Deletion

Removing a Black node breaks Black Height.

Requires complex "Double Black" fixing.

The Toolbox

Recoloring Flip Red/Black
Rotation Restructure Links

Rotation Types (O(1))

  • Single Rotation:
    Fixes straight lines (Left-Left / Right-Right).
  • Double Rotation:
    Fixes zig-zags (Left-Right / Right-Left).

Red-Black Tree Insertion

1. The Initial Insertion

  • Step 1: Standard BST Insert.
  • Step 2: New node (N) is RED.

Coloring Red preserves Black Height.

3. The Guarantee

Rebalancing is bottom-up.

  • Restores invariants in O(1) per level.
  • Total Time: O(log n).

2. Violation Cases (Parent is Red)

Case 1: Uncle is Red

Action: Recolor P, U, G.
Propagates up to Grandparent.

Case 2: Uncle Black (Zig-Zag)

Action: Rotation to form Line.
Transforms into Case 3.

Case 3: Uncle Black (Line)

Action: Rotate & Recolor.
Final Fix. P becomes Black root.

Red-Black Tree Deletion

1. Initial Deletion (BST Phase)

  • Step 1: Standard BST Delete (Z replaced by Y).
  • Step 2: Check removed node Y's color.
Red Y
Easy! No Rebalance.
Black Y
Hard! Black Height Broken.

3. Guarantee

  • Double Black is fixed or moved up.
  • Constant work per step.
  • Total Time: O(log n).

2. The "Double Black" Problem

Child X inherits Y's blackness, becoming "Double Black". We fix this by looking at sibling S.

Case 1: Sibling Red

Recolor & Rotate. Transforms to Black Sibling case.

Case 2: Sibling Black (No Red Children)

Recolor & Propagate. DB moves up to Parent.

Case 3/4: Sibling Black (Red Children)

Rotate & Final Recolor. Problem Solved.

2–3 Trees Overview

What is a 2–3 Tree?

A balanced search tree where every internal node has either 2 or 3 children.

  • Maintains Perfect Balance.
  • All leaves are at the same depth.

Why This Matters?

They provide the conceptual basis for balanced BSTs.

AA Trees simulate 2–3 trees using right-leaning links and levels.

2-Node
[ K1 ]
1 Key, 2 Children
3-Node
[ K1 | K2 ]
2 Keys, 3 Children

Key Properties

  • Keys are sorted inside nodes.
  • Ordering: Left < K1 < Mid < K2 < Right
  • Complexity: O(log n) for all ops.

Insertion in 2–3 Trees (Overview)

1. Insert into Leaf

New key is always inserted into a leaf node.

  • If leaf is 2-node → Becomes 3-node.
  • Simple, local operation.

2. Handling Overflow

If inserting into a 3-node, it becomes a temporary 4-node.

The Split Action:
  • Middle key moves UP to parent.
  • Left/Right keys form separate 2-nodes.
  • May propagate upward.

3. The Result

  • Tree remains Perfectly Balanced.
  • Height increases only when root splits.
  • O(log n) Complexity.

[ B ]
[ A ]
[ C ]

Visualizing the 4-node Split

Deletion in 2–3 Trees (Overview)

1. Delete from Leaf

  • Safe Case: Key is in a 3-node.
    Remove key → Becomes 2-node.
  • Hard Case: Key is in a 2-node.
    Removing creates a Deficient Node (Underflow).

3. Internal Nodes

Never delete internal nodes directly.

  • Replace with In-Order Successor (always a leaf).
  • Delete the successor from the leaf using leaf rules.

2. Fixing Deficiency

(a) Borrowing (Redistribution)

If sibling is a 3-node, take a key. Rotate through parent.

(b) Merging (Fusion)

If sibling is a 2-node, merge sibling + parent key + deficient node into a 3-node.

Warning: Parent might become deficient.

The Result

  • Height decreases only if root becomes empty.
  • Complexity: O(log n)

Disadvantages of Insertion

Red-Black Trees

  • Complexity: Many cases (recoloring + rotations) make algorithms complex.
  • Implementation: Harder to debug due to strict rules.
  • Overhead: Constant-time overhead from extra rotations.
  • Memory: Storage overhead for color bits.

2–3 Trees

  • Propagation: Node splitting may propagate up to the root.
  • Structure: Handling multi-way nodes is implementation heavy.
  • Pointers: More complex structure than binary trees.
  • Speed: Balancing rules make insertion slower in practice than simple BSTs.

Disadvantages of Deletion

Red-Black Trees

  • Double Black: More complicated than insertion due to "double-black" cases.
  • Restoration: Requires multiple rotations + recolorings to restore balance.
  • Maintenance: Hard to code, maintain, and debug.
  • Cache: Pointer-heavy operations → less cache-friendly.

2–3 Trees

  • Cascading: Merging or borrowing can cascade upward.
  • Complexity: Multi-node restructuring makes implementation complex.
  • Balance: Maintaining perfect balance requires many cases.
  • Pointers: More manipulation (3 children, temporary 4-nodes).

Introduction to AA Trees

Why a new tree?

We know Red-Black and 2-3 Trees are fast (O(log n)), but they are heavy to implement.

  • RBT: Many rotation cases, messy deletion.
  • 2-3 Trees: Multi-way nodes, upward splits.

"AA Trees were introduced to simplify this complexity."

What AA Trees Do

  • Simulate 2-3 Tree: But force all "horizontal" links to the Right.
  • Only 2 Operations: Maintain balance using just Skew and Split.
  • Less Cases: Removes many tricky RBT edge cases.

The Result

Simpler Code. Fewer Cases.
Still guarantees O(log n).

AA Tree Node Structure

Structure Breakdown

  • key: Data stored in the node.
  • left/right: Pointers to children.
  • level: Height proxy for balance.

Key Points

  • Level replaces RBT Color.
  • Horizontal links always Right-Leaning.
  • Leaf Level = 1. Null = 0.
C Implementation
typedef struct AANode { int key; struct AANode* left; struct AANode* right; int level; } AANode; // Function to create a new node AANode* createNode(int key) { AANode* node = (AANode*)malloc(sizeof(AANode)); node->key = key; node->left = node->right = NULL; node->level = 1; // leaf node return node; }
"Level is the core property ensuring balance."

AA Tree – Invariants

The 5 Rules

  • 1. Left Level = Parent Level - 1 No left horizontal links allowed.
  • 2. Right Level = Parent Level (or -1) Right horizontal links are allowed (simulates 3-nodes).
  • 3. No 2 Consecutive Right Links Prevents chains of right-leaning nodes.
  • 4. Leaves @ Level 1 Null pointers are treated as level = 0.
  • 5. Path Levels Non-Increasing Ensures balance and keeps height ≈ log n.

Summary

Invariants enforce 2–3 tree structure using simple rules.

  • Violations are fixed by skew and split operations during insertion/deletion.

Violation Handling

Violations are fixed immediately by Skew and Split operations.

AA Tree – Operations Overview

1. Search

Standard BST search using key comparisons.

Time: O(log n)

2. Insertion

Standard BST insert + Fix violations.

  • Fix using Skew and Split.
Time: O(log n)

3. Deletion

Standard BST delete + Fix imbalance.

  • Update levels, then Skew/Split.
Time: O(log n)

Balancing Tools

  • 4. Skew
    Right rotation. Removes left horizontal links.
  • 5. Split
    Left rotation + Level Up. Fixes consecutive right links.
  • 6. Level Update
    Adjusts level to match children after deletion.

Summary

A small set of simple operations maintains balanced BST properties efficiently.

Skew Operation – Concept

Purpose

Removes Left Horizontal Links.

  • Maintains the invariant: "No left horizontal links allowed".

When to Use?

If a node’s Left Child has the same level as the node itself.

Key Points

  • Only fixes left horizontal links.
  • Often followed by Split.
  • Simple, local O(1) operation.

How It Works

Right Rotation around the parent.

Skew Operation – C Code & Complexity

Time Complexity

O(1)

Skew is a single rotation, independent of tree size.

Summary

  • Skew is simple, fast, and local.
  • Fixes left horizontal link violations.
  • Essential step during insertion and deletion.
skew.c
AANode* skew(AANode* node) { if (node == NULL) return node; if (node->left && node->left->level == node->level) { // Right rotation AANode* left = node->left; node->left = left->right; left->right = node; return left; // new root of this subtree } return node; // no skew needed }

Maintains BST structure and AA Tree invariants locally.

Split Operation – Concept

Purpose

Removes Consecutive Right Horizontal Links.

  • Maintains the invariant: "No two consecutive right horizontal links allowed".

When to Use?

If a node’s Right Child and Right-Right Grandchild are at the same level.

Key Points

  • Fixes right horizontal link violations.
  • Often applied after Skew.
  • Ensures AA Tree remains balanced.

How It Works

Left Rotation + Level Increment.

Split Operation – C Code & Complexity

Time Complexity

O(1)

Split is a single rotation, independent of tree size.

Summary

  • Split is fast and local.
  • Fixes consecutive right links.
  • Works together with skew during insertion/deletion.
split.c
AANode* split(AANode* node) { if (node == NULL) return node; if (node->right && node->right->right && node->level == node->right->right->level) { // Left rotation AANode* right = node->right; node->right = right->left; right->left = node; right->level += 1; // increment level return right; // new root of this subtree } return node; // no split needed }

Maintains BST structure and AA Tree invariants locally.

Insertion – Concept & Cases

Overview

1. Insert key like in a standard BST.

2. Fix violations on recursion return:

  • Left Horizontal Link → Skew
  • Consecutive Right Links → Split

Step-by-Step

  1. BST Insert: Recursive placement.
  2. Skew: If left->level == level.
  3. Split: If right->right->level == level.
  4. Return: Update parent pointers.

Violation Cases

Case 1
No Violation
Case 2
Skew Only
Case 3
Split Only
Case 4
Skew then Split

Result

Tree remains balanced.
Operations stay O(log n).

Insertion – Code & Proof

Time Complexity Proof

  • Recursion: Follows height h → O(h)
  • Skew + Split: O(1) per node (local)
  • AA Tree Height: Balanced → h = O(log n)
Total Time = O(log n)

Key Takeaways

  • Skew/Split are constant time.
  • Recursion dominates the cost.
  • Guarantees O(log n) worst-case.
insert.c
AANode* insert(AANode* node, int key) { if (node == NULL) return createNode(key); // new leaf if (key < node->key) node->left = insert(node->left, key); else if (key > node->key) node->right = insert(node->right, key); else return node; // duplicate keys not allowed node = skew(node); // fix left horizontal link node = split(node); // fix consecutive right links return node; }
"Standard BST insert + Automatic Rebalancing"

Deletion – Concept & Steps

Overview

Delete key like a standard BST. Then fix invariants.

Violations to Fix:
  • Left Horizontal Links → Skew
  • Consecutive Right Links → Split
  • Level Decrease → Update Level

Result

Tree remains balanced.
Complexity stays O(log n).

Step-by-Step Process

1. BST Deletion
  • Leaf: Remove it.
  • One Child: Replace with child.
  • Two Children: Replace with successor.
2. Level Update

Set level = min(left, right) + 1.
Ensures consistency with children.

3. Skew & Split

Apply Skew (fix left links) then Split (fix right chains).

Deletion – Code & Proof

Time Complexity Proof

  • Recursion: BST search cost → O(h)
  • Fixes: Level update, Skew, Split are O(1) per node.
  • Height: h = O(log n)
Total Time = O(log n)

Key Takeaways

  • Involves local rotations + level updates.
  • Simpler than Red-Black Tree deletion (fewer edge cases).
delete.c
AANode* decreaseLevel(AANode* node) { int shouldBe = 1 + ((node->left ? node->left->level : 0) < (node->right ? node->right->level : 0) ? node->left->level : node->right->level); if (shouldBe < node->level) { node->level = shouldBe; if (node->right && shouldBe < node->right->level) node->right->level = shouldBe; } return node; } AANode* delete(AANode* node, int key) { if (!node) return NULL; if (key < node->key) node->left = delete(node->left, key); else if (key > node->key) node->right = delete(node->right, key); else { // key found if (!node->left && !node->right) { // leaf free(node); return NULL; } else if (!node->left) { // one child AANode* temp = node->right; free(node); return temp; } else if (!node->right) { // one child AANode* temp = node->left; free(node); return temp; } else { // two children AANode* succ = node->right; while (succ->left) succ = succ->left; node->key = succ->key; node->right = delete(node->right, succ->key); } } node = decreaseLevel(node); node = skew(node); if (node->right) node->right = skew(node->right); if (node->right && node->right->right) node->right->right = skew(node->right->right); node = split(node); if (node->right) node->right = split(node->right); return node; }

Time Complexity Analysis

1. Preliminaries

  • Height h is O(log n).
  • Simulates 2–3 tree: h ≤ 2 · log2(n+1)

Operations involve traversing path (O(h)) + local rotations (O(1)).

2. Insertion

  • Step 1 (Search): O(h)
  • Step 2 (Skew/Split): O(1) per level × h levels
Tinsert = O(h) = O(log n)

3. Deletion

  • Step 1 (Search): O(h)
  • Step 2 (Level Update): O(1) per level
  • Step 3 (Skew/Split): O(1) per level
Tdelete = O(h) = O(log n)

Key Insights

Height traversal dominates. Rotations are constant overhead.
Worst-case is guaranteed O(log n).

Applications of AA Trees

Database Indexing

Efficiently maintains sorted keys with O(log n) CRUD operations.

In-Memory Maps

Dictionaries storing key-value pairs with balanced search properties.

Priority Queues

Used where insertion and deletion frequency demands balanced performance.

Systems Programming

Easier to implement than Red-Black Trees in kernels or compilers.

Real-time Systems

Applications needing predictable, guaranteed O(log n) operations.

Disadvantages of AA Trees

1. Taller than Red-Black

Can be flatter (more spread out) but in worst cases slightly taller due to right-leaning restriction.
Result: Slightly slower search.

2. Memory Overhead

Requires an extra int level field per node.
Increases memory usage vs simple BST.

3. Still Complex

While simpler than Red-Black, Skew/Split logic is harder than standard BSTs or AVL trees (conceptually).

4. Adoption

Not widely used in standard libraries (STL uses Red-Black).
Deletion logic (level adjustment) is involved.

Conclusion

Why use AA Trees despite the disadvantages?

1. Simplicity

Removes complex Red-Black cases. Only 2 operations: Skew & Split.

2. Performance

Provides the same asymptotic O(log n) performance as industry standards.

3. Reliability

Deterministic balancing mimicking 2-3 Trees.

4. Education

Ideal for teaching balanced trees without the "implementation nightmare" of RBTs.

"AA Trees offer a powerful balance between Simplicity and Efficiency."
